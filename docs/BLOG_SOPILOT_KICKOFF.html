Product kickoff · conversational agents · discussion I

The Supervisor Ships

MCPlanner asked a research question: can a slow, smart supervisor make a fast, weak voice agent behave like an expert who follows the procedure? The answer came back with numbers — and, just as valuably, with a list of things that don't work. This is the discussion doc for turning the working subset into a product: SOP-based conversation agent management, with realtime voice as the first-class citizen.

The setup

Where this starts

A voice agent has one job that fights itself: be fast (sub-second, or the dialogue feels broken) and be right (follow the SOP, use the customer's real data, don't improvise policy). MCPlanner's answer was two systems and one pool: a fast on-path agent that only ever reads, and a slow off-path supervisor that predicts where the conversation is going and pre-stages what the agent will need.

The POC worked well enough to write up. But a research repo is a place where good and bad ideas live side by side, documented with equal affection. The product cannot afford that. So the first task of this discussion is triage: what did the experiments actually validate, what did they kill, and what did they never get to.

One more thing changed since we started: the outside world built the measuring stick. SOP-Maze (Meituan, 397 tasks distilled from 300K real customer-service API logs) shows frontier models failing at exactly the three things we target — route blindness in branching procedures, conversational fragility, and calculation under context. SAGE benchmarks graph-guided service dialogues; SOP-Bench covers tool-orchestration SOPs. When we claim the supervisor architecture fixes SOP-following, there is now a public yardstick to prove it on — and the benchmarks' failure catalogues read like our feature list.

◆ The product in one sentence
A system that turns a plain-text procedure into a versioned SOP graph with stage-bound prompts, then supervises live conversations against it — tracking state in the background, injecting the right instructions and the right retrieved data into the agent at every turn, without ever standing on the latency-critical path.

The drop list

What stays in the lab

These aren't guesses — each one was tried and measured. Writing them down is what stops them from sneaking back in under a new name during a design review.

Retired ideaWhy it died
live MCTS on the critical path 10–15 s, two-thirds of turn latency; removing it cost nothing once the pool carried the load
MCTS for retrieval prediction 33% recall@3 at ~25.6K tokens vs counting's 88% at zero — “habits, not hypotheticals.” Kept only as an optional cold-start planner

confidence gates & cosine floors real pool cosines cluster ~0.30 with 0.013 median gaps — there are no cliffs to gate on; 0% adoption of the fast path in practice
exact-key / query-hash cache model discards approximately-right prefetches — the 60% “≥1 shared doc” signal the pool monetizes
Thompson sampling exploration identical to decay+shrinkage (49.4% vs 49.4%): the true next action is revealed every turn, so this is online supervised learning, not a bandit
smaller live model to cut latency gpt-4o-mini was slower on the structured call and collapsed success 10/10 → 4/10; the LLM floor yields to streaming, not shrinking
“curated, high-relevance” framing actively harmful on closing turns — the agent answers about discounts while the user is saying goodbye
◆ One nuance worth keeping
Pondering isn't dead — it was mistested. At zero-pause simulator cadence, 98% of pondering runs were cancelled and it contributed 3% of pool fetches; with realistic think-time injected, its hit rate jumped to 86%. In production, real users pause. Ship it off by default, instrumented, and let per-tenant data decide — that's a flag, not a conviction.
The honest gap

The unbuilt half is the product

MCPlanner validated data prefetch (Milestone A) and never built instruction prefetch (Milestone B). The product you asked for — pre-built prompts bound to conversation stages, served just-in-time — is precisely Milestone B. We are not porting a finished result; we are building the missing half on top of a validated substrate.

That's a favorable position, for two reasons. First, the substrate generalizes: an instruction is just a PoolItem with kind:"instruction" — same pool, same TTL, same rerank, same honest framing. The POC already sketched this (schedule_instruction_prefetch exists; it's exact-match and unmeasured). Second, the realtime Avatar path already does a degenerate version of it every turn: it builds instructions live from the chosen action's must_say/must_not_say and injects them. The product upgrades that from “assembled from SOP fields at plan time” to “authored prompt blocks, versioned in a library, bound to stages, pre-staged by prediction.”

◆ Pre-commit the criteria now, like the research did
Instruction prefetch ships only if it beats the live-assembled baseline on: instruction hit-rate at lookup (target ≥70%, the production note's load-bearing bet), no regression in session success, and plan-turn p95 under the voice budget. The lab notes' open questions Q1–Q8 (granularity, trust model, wrong-instruction feedback loop, break-even economics) become the product's first experiment plan — with N≥20, not N=5.
The proposal

The product shape: four services, one pool

Everything the user-facing ask contains — SOP from plain text, stage-bound prompts, background tracking with live instruction delivery, a management UI, and production retrieval — falls out of four services around the same two-lane architecture the research locked.

◆ Tenancy is structural, not a checkbox
SOPilot is multi-tenant and multi-project from the first migration: tenant → project → SOPs, and everything hangs off that spine — prompt libraries, corpora, precedent traces, pool namespaces, dashboards, API keys. Two consequences worth locking now. First, the precedent store is tenant-scoped by construction: the counting predictor learns one tenant's conversational habits and must never leak them into another's predictions (it's also simply better prediction — habits are per-business). Second, isolation follows the pattern already proven in Context Graph's workspace pool: scoping enforced at the storage layer (Postgres row-level scoping + per-tenant Redis key namespaces), not in application if-statements. Projects give a tenant separate lifecycles per use case — the collections SOP and the sales SOP share a prompt library only when their owner says so.

SOP Studio — plain text in, governed procedure out

The POC's chat-to-build flow (build-turn + server-side patch merge, graph re-derived every step so you watch it grow) ports directly and gains three product-grade layers: batch ingestion (paste a policy document or upload a PDF, get a draft SOP to refine conversationally), a linter (the credit-card SOP's objection-loop pathology was an authoring bug — state-prereq gating, reachability and loop checks become publish blockers, not incident reports), and versioning with publish/rollback, because in production the SOP is a governed artifact someone signs off on.

The prompt library — stages get authored voices

Each SOP stage (agent action) binds to prompt blocks from a reusable library: role framing, stage instructions, must_say/avoid constraints, compliance boilerplate, escalation language — separately versioned from the SOP so legal can update the disclosure text without touching the graph. At runtime these are what instruction prefetch pre-stages.

Supervisor runtime — the background tracker

The locked no-tier-3 pipeline, portably re-plumbed: classify (cohort, state, mood) → SOP position update → empirical predictor over the tenant's precedent traces → prefetch data + instructions into the pool → cosine+dedup rerank on demand. Stateless workers; all session state in Redis; every decision audited to Postgres.

Operations Console — the research UI, grown up

The POC frontend already shows the right things (SOP graph position, blackboard contents, per-turn traces, hit-rate chips). The console keeps that per-session X-ray and adds the fleet view: the SLIs from the production note — effective hit rate, latency hidden, live-fallback rate, tier distribution, rerank p95 — per SOP, per tenant, from day one. The 70% instruction-hit-rate bet is either visible on a dashboard or it isn't real.

The mechanism

One turn, end to end (voice)

The realtime path is the part most products get wrong, and the part the POC already solved elegantly: run the voice model with manual turn control (turn_detection.create_response=false), so there is a supervisor window between “user stopped speaking” and “agent starts speaking” — and keep that window under half a second by making it contain zero LLM calls.

Two contract rules carry over from the research as non-negotiable: the injected context block is always labelled speculative (“ignore what doesn't fit; if nothing fits, reply naturally”), and the live agent never blocks on a supervisor product — a pool miss falls back to the stage's baseline prompt and gets logged as the SLI event that it is.

Production grade

Retrieval & persistence, for real

The POC's honest confession: the pool is a process-memory singleton, retrieval runs over three authored fixture corpora, and every non-RAG fetcher is a mock returning canned text. That was correct for research. For the product, this layer is where most of the porting work lives.

state

Postgres + pgvector replaces SQLite + sqlite-vec for everything durable: SOPs and versions, prompt library, precedent traces and embeddings, per-call audits, corpus documents. Redis holds the live session pool (TTL is native) and feeds the console's live views — this is what makes the runtime multi-worker and horizontally scalable, which the in-memory singleton structurally wasn't.

fetchers

A fetcher SDK replaces the mocks: MCP tools first (one connector standard, plays into the existing PolarTie stack), then REST/DB connectors. Declared per data-dependency in the SOP, with the POC's good bones kept: declared latency, TTL, idempotency flags — the scheduler still refuses to speculatively fire non-idempotent calls.

corpora

Manageable retrieval means corpora are tenant data with a UI: upload and chunk documents per corpus, bind corpora to SOP data-dependencies, re-embed on model change, inspect what any given turn retrieved and why. Embeddings computed at ingest, never on the turn path; per-turn cost is one query embedding + cosine math.

audit

Every fetch, pick, injection, and fallback lands in Postgres with its predictor source and outcome — the same discipline that let the research compute regret against an oracle becomes the product's analytics substrate, and the raw material the counting predictor learns from. The trace is the moat: every conversation makes the next one cheaper.

◆ Ecosystem note
The retrieval tier deliberately speaks MCP so a Context Graph / KnowledgeRag workspace can serve as a corpus or precedent backend where a customer already runs one — the supervisor's “query KG during the user's speech-time” lane was designed for exactly that shape. Optional, not a dependency.

The honest ledger

Where it can fail

simulator gap

Every headline number comes from LLM-simulated users — no ASR noise, barge-in, or topic whiplash. The architecture should transfer; the rates won't survive contact unchanged. Budget for re-measurement on real calls before quoting numbers to customers.

the core bet is unproven

Instruction prefetch — the product's center — was never measured in the research. It inherits a validated substrate, but “instructions behave like data in the pool” is a hypothesis with pre-committed criteria, not a result.

the latency you can't hide

The pool hides external-data latency, not model inference. In the text pipeline two live LLM calls (~4 s) dominated. Realtime speech-to-speech models mostly dissolve this — which is exactly why voice-first is the right wedge — but classify-quality on a fast model needs watching.

SOP authoring quality

The one SOP that failed (credit-card, 0/5) failed on graph pathology, not architecture. Customers will author worse graphs than we did. The linter isn't a nice-to-have; it's the difference between “their bug” and “our outage.”

cold start economics

The counting predictor is only as good as accumulated traces; a new tenant has none. Shrinkage helps (+12 pp), simulator-seeding helps, but the first weeks of any deployment run at the architecture's worst operating point — precisely when the customer is judging it.

crowded lane

VoiceAgentRAG, LTS-VoiceAgent, PASTE — the two-system pattern is converging independently in several labs. The defensible parts are the SOP governance layer, the precedent flywheel, and the management surface — not the prefetch trick alone. Speed matters.

◆ The honest reframe
Four of the six are about evidence and operations, not architecture. That's the healthy failure profile for a productization: the research de-risked the design; what remains is proving it where it's cheap to be wrong (harness, linter, dashboards) before customers prove it where it's expensive.
Sequencing

The build order

Ordered so that every phase ends with something demoable, and the riskiest unproven piece (instruction prefetch) lands on top of already-hardened plumbing rather than alongside it.

PhaseScopeExit criterion
P0 — substrate port planner core off SQLite: Postgres+pgvector, Redis pool, stateless supervisor workers, tenant→project→SOP scoping + auth enforced at the storage layer, fetcher SDK (MCP first), audit schema locked-config POC benchmarks reproduce on the new stack (same hit rates, rerank p95 ≤500 ms) with 2+ workers and 2+ isolated tenants running concurrently
P1 — SOP Studio document→draft-SOP ingestion + conversational refinement (port build-turn), linter (prereq gating, loop & reachability checks), prompt library with stage bindings, versioning/publish a non-author can go from a pasted policy doc to a lint-clean, published SOP with bound prompts in one sitting
P2 — realtime runtime voice adapter (manual turn control + session.update), text adapter, instruction prefetch behind its pre-committed criteria, pondering behind a flag with pause-aware scheduling live voice demo on 2 SOPs; instruction hit ≥70%; plan-turn p95 <500 ms; N≥20 ablation vs live-assembled baseline
P3 — console & proof fleet dashboards (SLIs), live session X-ray, corpus manager, trace explorer; SOP-Maze/SAGE harness runs; human A/B via the avatar rig the external-benchmark table exists and says what the marketing says
◆ The one-sentence version
The research already paid for the expensive lessons — counting beats search, pools beat caches, honesty beats confidence, and nothing smart belongs on the critical path. The product is those four lessons with a database, a linter, a prompt library, and a dashboard — plus one new bet, instruction prefetch, which we will measure the same way we measured everything else: criteria first.

Open items for this discussion: P0 stack confirmation (Postgres+pgvector+Redis vs. anything already standard in-house), whether SOP-Maze or SAGE is the first external target, and which two SOPs make the P2 demo.
