Spike scope · visualisation-first · config-management

A PolarTie config, as a dependency graph

The fastest way to prove the config-management direction is to show it: take a real PolarTie config.json and render it — in SOPilot's existing Studio — as a dependency graph, a status summary, and a validation report that catches the silent misconfigurations PolarTie's system can't. This page is the target mockup (built from a realistic sample config) plus the scope to build it.

◆ The one-line goal
Paste a robot's config.json → get a screen-shareable dependency graph, config validation, and logical prompt validation (the freeform prompt checked against the config — PolarTie's biggest pain), built almost entirely from components we already have, before any new backend. "Here is your config as a graph, with every tool, MCP server, knowledge base and dependency — here's what's mis-wired, and here's where your prompt promises things the config can't deliver."

Logical prompt validation — the biggest pain (LLM-checked)

The freeform prompt is PolarTie's default prompt system (SOPs are opt-in). An LLM reads it against the config and flags logical mismatches — the thing PolarTie engineers can't do today:

✖The prompt says "I can email you a summary of your enquiry" — but send_email is broken (no notification service). The agent will promise something it can't do.
⚠The prompt instructs "always reply in Spanish", but the config lists es/en and offers English — a possible contradiction to confirm.
⚠The prompt mentions "transfer to the baggage desk" — transfer is on and a baggage topic exists ✓, but it also mentions a "lost-property desk" with no matching topic.
✔No internal self-contradictions detected in the instructions.

The scope

◆ In / out for the spike
In: load a real config.json, transform it to a graph, render it (dependency graph + status + structural validation + logical prompt validation) in the Studio; read-only. Out (for now): editing, the guided wizard, RBAC, writing back to S3/Portal, the general engine abstraction. This is a read-only visualiser — the cheapest slice that proves the whole thesis.

1 · The configToGraph transform (the one new piece)

A pure function config.json → {nodes, edges, problems}. Mapping rules:

From the configBecomes
the robot (display_name, gpt_model, voice, language)the root Agent node
each enabled entry in tools (enabled + non-empty description)a tool node (disabled ones dimmed or hidden)
each mcp_servers[]an MCP node (its tools resolved live via list_tools)
each knowledge_base[] (simple / lightrag)a KB node → edge to its backend (OpenSearch / Postgres)
each transfer_topics[]a topic node ← edge from the transfer tool
notification_service_url, lightrag.postgres, opensearch_endpointdependency nodes that tools/KBs point to (red if required-but-missing)

2 · The validator — two prongs

Structural (formal, cheap): the collect-problems pattern SOPilot's linter already uses, seeded with real PolarTie foot-guns — send_email/send_sms require notification_service_url; a LightRAG KB requires lightrag.postgres; a simple KB requires opensearch_endpoint; transfer enabled wants ≥1 topic. Each becomes a red node/edge + a validation line.

Logical prompt validation (LLM — the headline): the freeform prompt is PolarTie's default, so this works with or without SOPs. An LLM reads the prompt against the resolved config and flags the mismatches PolarTie engineers can't catch today — the prompt promising a capability that's off/broken, contradicting a setting, or referencing something unconfigured. This is the biggest pain, and it's squarely SOPilot's LLM-native wheelhouse.

Net new code ≈ one transform module + one validator + one thin Config view. Everything visual is existing Studio.

4 · Change history (fast-follow, not in the first cut)

"Track changes over time" reuses SOPilot's append-only versioning (the SopVersion pattern): store each config as an immutable version, show a version list + a field-level diff (what a change touched, and which graph nodes/edges it moved). Small once the graph exists; scoped as the next step, not the demo.

5 · Data we need & acceptance

Data: one or two real robot config.json files (or we use the representative sample above). Done = paste a config → the dependency graph renders, the status card is correct, and the validator flags the missing-dependency case with a red edge — screen-shareable in the existing Studio, no new backend.

◆ Why this is the right first move
It's a few days of mostly-glue work, it produces the single most convincing artefact for the PolarTie conversation, and every piece it builds — the config→graph transform, the validator, the Config view — is load-bearing for the full product later. We prove the thesis before committing to the wizard, the RBAC, or the engine abstraction.
