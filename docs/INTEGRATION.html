Orientation

The integration at a glance

◆ Choose your integration depth (D-9)
Every project — and even every single session — can run both subsystems (default), sop only (procedure tracking + governed prompts, plain live lookups), or retrieval only (prediction + pre-staged context for your agent). Set it at project creation, change it with a PATCH, or override per session at start.
Step 1

Bootstrap & auth

# tenant — returns the API key EXACTLY ONCE; store it in a secret manager
curl -X POST $BASE/admin/tenants -H "X-Admin-Token: $ADMIN_TOKEN" \
  -H 'Content-Type: application/json' -d '{"slug":"acme","name":"Acme"}'

# project (subsystems: "both" | "sop" | "retrieval" | "" = deployment default)
curl -X POST $BASE/admin/projects -H "Authorization: Bearer $KEY" \
  -H 'Content-Type: application/json' -d '{"slug":"collections","subsystems":"both"}'

Every project-scoped call carries two headers: Authorization: Bearer sop_… and X-Project: collections. Keys are tenant-scoped — cross-tenant access returns 404, never data. Expect: 401 bad key · 404 unknown object/project · 409 state conflict · 422 validation or lint failure (with problems[]) · 429 tenant turn quota.

Steps 2–3

Author & publish an SOP

Paste the written procedure — or upload the PDF — and refine the draft conversationally; publish is gated by a linter that refuses broken graphs.

# document → draft SOP (also: POST /sops/ingest-file with a PDF, 2 MB cap)
curl -X POST $BASE/sops/ingest -H "$AUTH" -H "$PROJ" \
  -d '{"text": "<the procedure text>", "name_hint": "Overdue Invoice Recovery"}'
# → {"id", "definition", "lint": {"problems": [], "publishable": true}}

# refine (stateless — you hold the working definition, save when happy)
curl -X POST $BASE/sops/build-turn -H "$AUTH" -H "$PROJ" \
  -d '{"history":[{"role":"user","content":"add a SmallBusiness cohort"}], "current_definition": {...}}'

curl -X PUT  $BASE/sops/$SOP_ID  -d '{"definition": {...}}'   # save new draft version
curl -X POST $BASE/sops/$SOP_ID/publish                        # 422 + problems[] if lint fails

Approved wording: prompt blocks

Compliance/legal language lives in a separately-versioned library and is bound to SOP stages; bindings resolve to published versions and are pinned when a session starts — a mid-call wording change never lands mid-call.

curl -X POST $BASE/prompt-blocks -d '{"name":"compliance.recording","kind":"compliance","content":"This call may be recorded…"}'
curl -X POST $BASE/prompt-blocks/compliance.recording/publish
# bind: add "compliance.recording" to an action's prompt_blocks[] and re-publish the SOP
◆ Two authoring rules that bite
Mark every mutating data dependency "idempotent": false — the prefetcher trusts the flag and will never speculate on it. And keep at least one non-terminal user_state: an all-terminal vocabulary is a lint error because per-turn tracking degenerates (learned in live testing, now enforced).

External data over MCP

# store the credential once per tenant (encrypted at rest; reads return names only)
curl -X PUT $BASE/secrets -H "Authorization: Bearer $KEY" \
  -d '{"name":"kr_api_key","value":"<credential>"}'

# then in the SOP's data_dependencies:
{"name":"kb","kind":"mcp","idempotent":true,
 "config":{"server":"https://kg.example.com/mcp","tool":"query_knowledge_graph",
            "query_arg":"query","auth_secret":"kr_api_key","auth_header":"X-API-Key"},

Step 4a

Text channel

SESSION=$(curl -X POST $BASE/sessions -d '{"sop_id":"'$SOP_ID'"}' | jq -r .session_id)

# one call per user message — classification, planning, and the reply all happen inside
curl -X POST $BASE/sessions/$SESSION/converse -d '{"user_message":"Hi, about my invoice…"}'
# → {"reply", "terminal", "classification", "turn": {chosen_action, picks, consume_stats,
#    instruction_hit, prompt_text, context_block, ...}, "total_ms"}

# when terminal (or your own hangup logic):
curl -X POST $BASE/sessions/$SESSION/outcome -d '{"outcome":"success"}'
curl -X POST $BASE/sessions/$SESSION/end

Say reply to the user; render the turn object in your own tooling if you want the X-ray. instruction_hit: true means the reply was pre-drafted by the supervisor and served verbatim — no generation latency on that turn. Callers that run their own classifier can use POST /sessions/{id}/plan-turn instead and receive the instruction payload without a generated reply.

Step 4c

Retrieval-only: your agent, our context

Already have an agent stack? Run sessions with "subsystems": "retrieval" and take only the pre-staged context.

curl -X POST $BASE/sessions -d '{"sop_id":"'$SOP_ID'","subsystems":"retrieval"}'
# per turn: converse or plan-turn → use turn.context_block in YOUR prompt; prompt_text is empty
◆ Keep the framing
context_block begins with "speculatively pre-staged — may or may not fit this turn". That wording is measured behavior (it prevents agents from forcing stale data into closing turns). Insert the block verbatim; never re-label it as verified.

Observability

EndpointWhat you get
GET /metrics/summary?days=Nthe SLIs: pre-staged availability (target ≥70%), caller-felt waits (<10%), instruction turn-hits vs the 70% claim gate, selection p50/p95, supervisor lag, outcomes
GET /sessions/{id}/fetchespermanent per-session prefetch audit — every fetch/draft marked served · unused · pending · error
GET /sessions/{id}/poolthe LIVE pool only (cleared at session end; items TTL out — by design)
Contract

The five integration invariants

#RuleWhy
1Nothing blocking between the user's utterance and converse/voice-turnthe runtime owns the latency budget; anything you add, the caller hears
2Insert context_block verbatim, framing includedthe honest label is measured behavior, not boilerplate
3idempotent:false on every mutating dependencythe prefetcher trusts the flag; a speculative booking is an incident
4Report outcome on every session you canterminal rewards are the predictor's training signal — your hit rates depend on it
5Tenant keys shown once, stored hashed; OpenAI key server-side onlybrowsers only ever hold short-lived realtime secrets
