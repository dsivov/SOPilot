Landing SOPilot in a Real Voice Agent

PolarTie already ships a production voice agent — OpenAI Realtime, a Python brain, an MCP tool loop, and its own scenario-driven test harness. That is the best possible place for SOPilot to prove itself, because we don't have to build the voice stack to demonstrate the supervisor. This is the discussion doc that maps the integration seams from their code (file and line), and proposes a two-stage plan: tests first, then product.

The setup

Why this is the right host

The customer ask was to find integration points into PolarTie's voice agent and come up with a plan. The finding that governs everything below: this is not a greenfield build. PolarTie's agent already runs the exact pipeline SOPilot was designed to steer — an OpenAI Realtime session, a per-turn boundary where a supervisor could inject instructions, and a standard MCP client that turns any external tool server into model-callable functions. Three separate reads of the codebase converged on the same three seams.

The architecture is a thin C++ worker that only launches the agent and pipes audio/config, and a Python brain that owns all the Realtime logic. Everything SOPilot needs to touch lives in that Python (src/voice-agent/). And it ships with a real test harness (tests/voice_agent_harness/) that drives a caller through a scripted call and asserts on transcripts, subtitles, events and audio — which means we can prove SOP-following behavior before touching their production loop.

◆ The plan in one sentence
Stage 1: add harness scenarios that stand SOPilot up behind the two zero-code seams (mcp_servers and prompt) and prove procedure-following with the harness's existing assertions. Stage 2: wire SOPilot's per-turn steering into the real Realtime loop at the one hook their code already exposes, with the mode (advisory vs. gated) chosen per procedure.

The findings

The three integration seams

SOPilot can enter the PolarTie agent at three distinct points. Two of them require no code change on their side (they're config); the third is a small, well-precedented hook in their Python loop. Ranked by how much of SOPilot they expose.

Seam A  mcp_servers — SOPilot as a tool server (zero code)

Their MCP client is a full standard implementation on top of FastMCP: connect and initialize, tools/list (mcp_session.py:356), tools/call (mcp_session.py:181), result-to-model, ping/reconnect, even sampling. Any tool a server advertises surfaces to the Realtime model as a function named mcp_<toolname> (tool_mcp.py:62). To plug SOPilot in, the operator adds one entry to config.mcp_servers — {url, authorization} (session_config.py:72) — and the model can call SOPilot's SOP tool. This is the exact shape D-11 (the SOPilot MCP-server surface) was committed to.

◆ The four rules a SOPilot MCP endpoint must obey
(1) Expose Streamable HTTP — FastMCP infers transport from the URL string; an http(s):// URL is Streamable HTTP. (2) Auth is a single authorization string (e.g. "Bearer …"). (3) Avoid the six reserved polartie_* tool names (pt-ai-agent-mcp/.../defines.py:5) — everything else "just works" as a normal tool. (4) Don't name input params pt_client_context or image_jpeg_base64 unless you want their special injection behavior (they're stripped from the schema shown to the model).

Seam B  prompt — SOP encoded in the system instructions (zero code)

The prompt field (session_config.py:132) becomes the system instructions via generateGptInstructions() (master.py:976). This is the cheapest possible integration — a SOP compiled into prompt text — and the right shape for Stage 1's deterministic tests and for a first no-infrastructure pilot. It's static per session, so it can't do per-turn steering or live retrieval, but it proves the procedure logic end-to-end.

Seam C  per-turn session.update{instructions} — live steering (small hook)

This is the seam that unlocks the full supervisor. There is a real per-turn boundary and a real injection primitive:

◆ There's already a working precedent for exactly this
doUpdateInstructions() (master.py:347) already calls sendUpdateInstructions(generateGptInstructions()) on a 60-second timer to refresh time-of-day and caller context. SOPilot's steering is the same call — moved from a coarse 60 s poll to a per-turn hook, with the text coming from SOPilot's /voice-turn instead of a static rebuild. We are not inventing a mechanism; we're re-timing one they already run.
The design call

Which seam does which job

The two live seams are not competitors — they carry different traffic. MCP is the clean seam for tool calls (the model asks SOPilot something and gets structured data back). Per-turn session.update is the seam for steering (SOPilot decides how the agent should behave this turn, before it speaks). SOPilot's own architecture already separates these — /plan-turn is retrieval-only, /voice-turn is per-turn steering — so the mapping is one-to-one.

MCP seam — request/response

Best when SOPilot answers a bounded question the model chose to ask: look up a fact, fetch the current step's data, validate an entry. Fully closed loop — tools/call → content → function_call_output → response (mcp_session.py:181 → master.py:467). Degrades gracefully: a slow/unavailable endpoint retries 3× / 5 s then the call proceeds (multi_mcp.py:69). Zero code on their side.

session.update seam — steering

Best when SOPilot must shape the reply the model is about to give — inject the classified stage's prompt block, the speculative retrieved context, the must-say / avoid constraints. This is the supervisor pattern: it runs in the window between "user stopped" and "agent speaks," and (per SOPilot's locked design) contains no LLM call on the critical path. Needs the small hook at master.py:927.

◆ One caveat the code makes explicit
Default turn detection is semantic_vad with create_response: True (gpt_voice_assistant.py:147) — meaning OpenAI's server auto-creates the response the instant the user stops. To reliably steer before the model answers, Stage 2 must flip to client-driven responses (setTurnDetection(createResponse=False)) so the supervisor window actually exists. This is the single most important behavioral change on their side, and it's a one-line flip with an existing setter.
◆ Prod-engineer feedback (round 1) — make the hook itself an MCP extension
Reviewing this doc, PolarTie's prod engineer proposed a cleaner Stage-2 boundary: rather than SOPilot-specific code at master.py:927, generalize the per-turn hook into a PolarTie MCP extension. If a connected MCP server advertises a method with an agreed name / metadata, PolarTie automatically enables the supervisor behavior — flips to client-driven turns (setCreateResponsePending(False)) and calls that method every turn. Then Seam C collapses into Seam A: SOPilot connects as a pure MCP server and per-turn steering "just works" — add the server, done. Internalizing the loop (the master.py hook) becomes a later latency optimization, not the integration path. This mirrors their existing reserved polartie_* tool + progress-notification extension pattern — so the whole integration, both stages, is config-only until we choose to optimize. One implementation note they flagged: setCreateResponsePending(False) must not conflict with the loop's other ...(True) calls.

Which mode SOPilot runs is per procedure, not global. An info-desk procedure (PolarTie's help-desk shape, and AENA's) runs advisory — SOPilot suggests, the model stays fluent. A compliance procedure (identity checks, disclosures) runs gated — the SOP's must-say/avoid become hard constraints. Both are the same injection mechanism; only the strictness of the injected text changes.

The full picture

One prod-grade architecture

Three codebases meet here — PolarTie (the voice agent), SOPilot (the supervisor), and Context Graph (the shared LightRAG knowledge tier). The clean production design makes every cross-codebase link an MCP boundary, so components compose without either side reaching into the other's internals. PolarTie has access to all three repos, so where each piece runs is a deployment choice, not a constraint — the contract stays the same.

◆ One turn across the whole system
(1) Caller speaks → PolarTie Voice Agent (OpenAI Realtime). (2) At the turn boundary the supervisor extension calls sop_guidance on the SOPilot MCP server. (3) That reaches the SOPilot backend (in-process in prod, HTTP :8100 in the first cut): the intake router picks the SOP, the supervisor plans the turn against the pool + Postgres. (4) If knowledge is needed, SOPilot's fetchers query Context Graph over MCP. (5) SOPilot returns the stage steering → PolarTie applies it as session.update and the agent speaks. (6) The turn outcome flows back to SOPilot's precedent store — the flywheel that makes the next call cheaper. Every numbered hop between repos is MCP; nothing reaches across a codebase boundary by any other means.
◆ How SOPilot picks — and switches — the SOP
The caller doesn't choose a procedure; SOPilot's intake router does, from the words. Picking: on the first utterance that states a need, one cheap router call routes the session (verified: "where can I park?" → the parking/wayfinding SOP on turn 1). A pure greeting defers (one intake reply), and the very next turn is forced to the closest procedure — so routing costs 1 turn, at most 2. An explicit sop_id from the integration (IVR / deep link) skips the router entirely. Switching mid-call: on the voice path the switch router now runs every turn, in parallel with classify/retrieval (no added latency), judged on the latest utterance — so if the caller starts on parking and then asks about a flight, SOPilot re-routes to the flight SOP on that turn and re-plans it there, while same-topic follow-ups stay put (verified: no over-switching). The speech-to-text can be garbled — the router is built to route on the best plausible reading, not perfect clarity.
The test surface

What the harness can drive & assert

A harness scenario is a Python file exporting async def run(ctx) (run_scenario.py:948), optionally with DESCRIPTION, TIMEOUT_SECONDS, MODEL module globals. The ctx object (ScenarioContext, run_scenario.py:280) is the whole API: it starts a session with a config dict, synthesizes caller speech, and then waits on / asserts against everything the agent emits. This is enough to test SOP-following without any mocking.

drive the caller

start_session(model, config) (config takes prompt + mcp_servers directly, :336); say(text) synthesizes caller TTS and appends trailing silence so VAD fires (:373); audio_silence, audio_wav, json_message (simulate a UI button, :361).

observe the agent

wait_transcript(role) and wait_transcript_text_matches(pattern, role) (:535/:547) — the primary lever; roles are agent / client / system. Plus subtitles (wait_json_message_text_contains), robot audio, and ptEvents.

assert (hard-fail)

assert_transcript_text_matches(pattern, role) (:673), assert_pt_event_not_seen("mcp_error") (:640), assert_robot_audio_non_silent() (:698), assert_not_seen("media_eof"). Assertions raise on first failure, so ordering them along the procedure yields a hard stop at the first violated step.

the one blind spot

There is no direct "tool was called" signal. MCP compliance is observed indirectly: through the agent's transcript reflecting the tool result, the mcp_error ptEvent on failure, or — for exactness — by pointing mcp_servers[].url at a mock we control and inspecting its logs. Plan for behavioral assertions, not tool-call spies.

◆ Repeatability levers the harness gives us
LLM output is nondeterministic; the harness offers two mitigations. Use tolerant regexes and generous per-step timeouts, and — if a scenario needs to be truly deterministic — openai_base_url (session_config.py) can point OpenAI at a mock. To assert strict step ordering rather than mere presence, read ctx.transcript_records("agent") and check the matching lines' timestamps are monotonic. Note every conversational scenario needs OPENAI_API_KEY (both for the agent and for ctx.say TTS).
◆ Two prod-team pointers for the test rig
Cost: gpt-realtime is expensive for automated tests that don't need much intelligence — override to gpt-realtime-mini (our Stage-1 scenarios already pin MODEL = "gpt-realtime-mini"). Real voice: PolarTie already has a path to feed real recorded voice into the agent (and the harness streams a real WAV via ctx.audio_wav) — worth using for the supervisor case specifically, because synthesized TTS is unrealistically clean; real accented / noisy audio is exactly where the transcript-reliability risk above shows up.
Stage 1

Tests — prove SOP-following in their harness

The goal of Stage 1 is a green scenario suite in tests/voice_agent_harness/scenarios/ that demonstrates the PolarTie agent following a multi-step SOPilot procedure — using only the two zero-code seams and the harness's existing assertions. Nothing in their production loop changes; this is the safe, reversible proof.

◆ First results — Seam B baseline, 4/4 AENA procedures green
We built the Stage-1 scenarios against the real AENA / Málaga info-desk SOPs and ran them through PolarTie's own harness (gpt-realtime-mini, prompt-only, no live SOPilot). All four — lost luggage, flight/check-in, ground transport, airport services — passed, 5 assertions each: the agent progressed every procedure in order (identify / answer → provide / route → close), with 17 s+ of non-silent audio and no dropped call. It also showed the SOPs' answer-first shape — e.g. on transport it offered the train, then adapted when the caller asked for the bus. This is the deterministic baseline the MCP path (S1b) and Stage-2 steering must match or beat; the scenarios are now the regression suite. Sample (lost luggage): "which airline and flight were you on?" → "please head to the baggage office for Ryanair" → "you're welcome — have a good day." Two repro notes: on SELinux-enforcing hosts pass --security-opt label=disable via the runner's extra-args env; and the first container after a docker build can exceed the harness's fixed 10 s ready-timeout on the cold LightRAG import — just re-run warm.
◆ S1b — SOPilot-as-MCP, 4/4 AENA procedures green (the real integration)
We then built the SOPilot MCP server (sopilot/mcp_server.py, FastMCP · Streamable HTTP) — the D-11 entry tool sop_guidance — and pointed PolarTie's mcp_servers at it. All four procedures passed, and the transcripts prove the loop is real: the agent emits [mcp_sop_guidance] Invoked by Agent each turn, SOPilot's intake router auto-routes to the right AENA SOP, and the agent follows the returned steering — with zero mcp_error across ~40 tool calls. The tell that guidance actually flowed: the S1b prompt carries no SOP content, yet the agent produced the lost-luggage SOP's exact greeting ("Do you prefer to speak in English or Spanish?") — it could only have got that from the tool. One honest nuance: on flight_info the agent hit the SOP's real VerifyDetails stage ("confirm the departure time") that our 3-turn script didn't leave room to complete — evidence the SOP is being followed faithfully, and a note that fuller scripts should mirror each SOP's stage count. Net: the pure-MCP integration the prod engineer proposed works end-to-end today.
Stage 2

Product — wire the supervisor into the live loop

Stage 2 turns the tool-seam integration into the full supervisor by adding per-turn steering at the one hook their code exposes. This is where SOPilot stops being "a tool the model can call" and becomes "the supervisor deciding how the agent behaves this turn."

The change is localized and mirrors their existing 60-second refresh timer, so it sits inside a pattern their code already runs:

Wiring stepWhat SOPilot does
Session start → POST /sessions bind the call to a SOP (or the intake router picks one); returns the mode (advisory / gated) for this procedure
Flip to client-driven turns once (gpt_voice_assistant.py:1100) opens the steering window so the server doesn't auto-answer before SOPilot weighs in
Each turn at master.py:927 → POST /voice-turn returns the steered instructions (classified stage's prompt block + speculative retrieved context, labelled speculative); apply via sendUpdateInstructions, then setCreateResponsePending
Turn end at master.py:878 → POST /outcome feeds the precedent store so the next call is cheaper — the flywheel SOPilot's prediction runs on
Call end → POST /end at the shutdown handler (master.py:1294) closes the SOPilot session, flushes audit
◆ Why this is a small, safe change on their side
Every primitive already exists and is exercised in production: the hook (on_..._speech_stopped), the injector (sendUpdateInstructions), the turn gate (setTurnDetection / setCreateResponsePending). We add HTTP calls to SOPilot and flip one turn-detection flag. The MCP seam from Stage 1 stays in place for tool traffic. If SOPilot is unreachable, the fallback is the agent's existing baseline prompt — the same graceful degradation their MCP retry logic already assumes.
◆ Stage 2 — built & validated (the supervisor extension, pure-MCP)
We implemented the prod engineer's design: a generic supervisor MCP extension in PolarTie plus a reserved tool on the SOPilot side — no per-integration code. SOPilot (mcp_server.py) exposes polartie_ai_agent_supervisor (a SOPILOT_MCP_MODE gate can serve it alone). PolarTie (6 small edits): a new reserved name in pt-ai-agent-mcp; mcp_session.py detects the tool at connect and keeps it hidden from the model; multi_mcp.py gains hasSupervisor()/callSupervisor(); gpt_voice_assistant.py fires a new onUserTranscriptCompletedCb at the transcript-completed event (the correct async trigger — the transcript lags speech-stopped); master.py flips to client-driven turns when a supervisor is present and, each turn, calls the supervisor → sendUpdateInstructions → releases the turn. End-to-end result: with a prompt containing no SOP and no instruction to call any tool, the agent still followed the AENA lost-luggage SOP — producing SOP-specific facts like "the Ryanair baggage office in front of belt 38" — with 0 model tool invocations (the extension drove it), client-driven turns confirmed in the log, and 0 mcp_error after making the supervisor tool degrade gracefully on any backend hiccup. So Stage 1's model-driven MCP path and Stage 2's automatic per-turn steering both run today; the remaining optimization is the in-process mount to shave the extra hop's latency.
The shared substrate

The LightRAG tie-in — we already speak their retrieval

PolarTie's knowledge base runs on LightRAG — the same engine as Context Graph, which SOPilot already integrates with for the AENA deployment. That's not a coincidence to exploit later; it's a shortcut available now. The session config carries a first-class lightrag field and a knowledge_base list (session_config.py), so the retrieval tier the customer already populates is reachable from the same place SOPilot's corpora live.

◆ What this buys the integration
SOPilot's supervisor was designed to "query the knowledge graph during the user's speech-time" — its async lane speaks MCP precisely so a Context Graph / LightRAG workspace can be a corpus or precedent backend. Because PolarTie already runs LightRAG, SOPilot's retrieved-context injection can draw on the customer's existing knowledge base rather than a duplicated store. The tie-in is optional (SOPilot can carry its own corpora), but where the customer already has the data indexed, we don't re-index it — we point at it.

Their MCP client also advertises a sampling capability (mcp_session.py:285), so a SOPilot MCP tool could request LLM sampling through the session's own key if it ever needed to — available, not required. Worth noting only because it widens what the tool seam can do without new infrastructure.

The honest ledger

Constraints & unknowns

the auto-response default

Steering only works if the server isn't already answering. Stage 2 depends on flipping create_response to false (gpt_voice_assistant.py:147) — a one-line change, but one that alters turn timing and needs a careful A/B against their current UX. This is the change to validate first, on one procedure.

no tool-call spy in the harness

Stage 1 can't directly assert "SOPilot's tool was called" — only that the transcript reflects it and mcp_error didn't fire. For exact tool-invocation coverage we point mcp_servers[].url at a mock we control. Plan the assertions behaviorally from the start.

nondeterministic transcripts

Realtime output varies run to run. Scenarios need tolerant regexes, generous timeouts, and — where a test must be stable — the openai_base_url mock hook. Expect occasional flakes on tight assertions; design for presence-and-order, not exact wording.

per-turn latency budget

The steering window sits on the critical path between "user stopped" and "agent speaks." SOPilot's design keeps zero LLM calls there, but the round-trip to /voice-turn must fit the voice budget. Measure p95 on their infra, not ours — network placement matters. The prod engineer expects this plug to raise response latency meaningfully — treat it as a must-measure, not a footnote.

transcript reliability & async timing

Two prod-team facts that undercut naive transcript-based steering. (1) Timing: the "client finished speaking" event fires significantly earlier than the completed transcription — OpenAI's transcript runs asynchronously and slower than the realtime model, so at the master.py:927 hook the final transcript often isn't ready yet. (2) Quality: transcript and realtime voice run independently under the hood; the text SOPilot sees can be garbage (accents, poor speech) while the realtime model understood the audio perfectly. Implication: SOPilot must not assume a clean, ready transcript at the hook — options are to wait for/settle the transcript (costs latency), swap in a stronger transcription model when a supervisor is attached, or steer on turn/state rather than verbatim text.

whose code changes, and when

Stage 1 is entirely our scenarios + a SOPilot endpoint — no PolarTie code. Stage 2 needs a small change in their master.py. That's a coordination point, not a technical risk, but it sets the sequencing: prove everything at the tool seam before asking for the loop hook.

config delivery path

Production prompt/mcp_servers come from PolarTie's scheduler (scheduler_connection.cpp:296/:310), not a local file. Standing SOPilot up for a real call means getting the MCP entry into their backend config — an operational step to confirm with them, separate from the harness (where we control the config dict directly).

◆ The reframe
Nearly every risk here is about evidence and coordination, not feasibility. The mechanisms all exist in their code and are exercised in production. That's the healthy profile for an integration: the host already does the hard parts; what remains is proving our layer where it's cheap to be wrong (the harness) before it runs on real calls.
