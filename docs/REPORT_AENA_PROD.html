An AI Agent for the Airport Help Desk

AENA gave us ninety days of real conversations from the Málaga airport information desk. We used them to build an AI agent that speaks the desk's own procedures, answers from a live knowledge base, and was tested against what the human agents actually said in those same conversations. This report contains the results and our recommendation for production.

The opportunity

The use case

Málaga airport's information desk answers thousands of short, urgent questions: where do I check in, my suitcase didn't arrive, where do taxis leave from. The median call is 37 seconds — one question, one answer, in Spanish, English or French. AENA wants an AI agent that can answer these calls with the desk's real knowledge, at conversation speed, around the clock.

The raw material for this POC is unusually good: 90 days of actual desk conversations (17 April – 15 July 2026), recorded and transcribed. Nobody had to guess what travellers ask or how good answers sound — the data shows both.

The system

What we built

1 · Procedures learned from the desk's own calls

For each major topic we analysed the cleanest conversations and wrote down the procedure the human agents demonstrably follow — then converted each document into a machine-checkable conversation flow. Purple boxes are what the agent does (in order); blue boxes are what the traveller's side of the conversation looks like; green/red end the conversation. The amber dots mark steps that use live knowledge.

Lost & delayed luggage

The clearest procedure in the data: identify the airline (each one uses a different handling company), then send the traveller to the right desk with exact directions — offices, belts, reference codes.

Flight & boarding information

The highest-volume topic (29% of all calls): which counter, when check-in opens, where to board — connected to live flight data when AENA provides the feed.

Transport, parking & wayfinding

Nearly a third of calls combined: buses, trains, taxis, parking and “where is…” questions — every answer ends with directions.

2 · The knowledge base — served by Context Graph

Every concrete fact the human agents repeatedly gave — Ryanair check-in is around counter 363; the Ground Force baggage office is past belts 36-37; special assistance is the yellow counter near 301 — was distilled into 111 facts and loaded into Context Graph, the knowledge-retrieval server our team already runs in production. 3 · The agent

The pieces combine into one agent, managed by our SOPilot platform: a caller speaks; the system recognizes which procedure fits (or asks politely when the opening is just a greeting); the answering model — a fast, inexpensive model of the class used for real-time voice — receives the relevant procedure and freshly retrieved knowledge, and answers in the caller's language. Meanwhile, in the background, SOPilot tracks the conversation against the procedure, pre-fetches knowledge it predicts will be needed next, and records a complete audit trail of every answer, every retrieval, and every routing decision — the operational visibility a production service needs.

4 · The agent learns from every conversation

Each finished turn is stored as a compact experience record: the situation (what the traveller expressed, in which mood), what the agent did, and how the conversation ended. From these records the platform learns, per topic, what typically comes next — and uses it to fetch knowledge from Context Graph before the traveller asks. That anticipation is why the recommended configuration answers ~40% faster than an agent that must look everything up on demand: by the time the caller finishes their sentence, the answer material is usually already staged.

During this POC the loop was live: the test rounds themselves trained the prediction, which is measurable in the response times above. Operationally this means two things for production: the agent will keep improving on real Málaga traffic without intervention, and tests should run in a separate workspace so simulated conversations never mix into production learning (the platform isolates workspaces natively — this costs nothing).

The method

How we tested it

We did not invent test questions. Each test conversation replays a real traveller's need from a held-out recording (never used to build anything): a simulated traveller opens with the real first sentence and pursues the real goal. The benchmark for a good answer is what the human desk agent actually provided in that recorded conversation.

Three agent configurations faced the same 36 travellers under identical conditions — same answering model, same knowledge, same judge:

Each conversation is scored by an independent evaluator on three questions: did the agent provide the essential information the human agent provided (coverage)? did it give the concrete location/number when one exists, rather than "please check the screens" (concreteness)? and how satisfied would this traveller be (satisfaction, 1–5)? We also measure response time for every reply.

◆ The three findings that matter
1 — The procedures are the treasure. In a controlled pairing (identical agents, the procedures the only difference), the mined procedures alone lifted coverage by 14 points and concreteness by 27 points, at no latency cost. The knowledge extracted from AENA's own recordings is the single most valuable asset of this POC.

2 — SOPilot delivers that quality at conversation speed. Same quality class as the procedure-armed agent (within measurement noise), ~40% faster at the median — because it pre-fetches knowledge in the background — and it is the only configuration a production service can actually operate: automatic topic routing, conversation tracking, per-answer audit trail.

3 — The remaining ceiling is data we don't have yet. The most common shortfall in every configuration is operationally dynamic information — today's counter for this flight — which lives on AENA's live screens, not in any static knowledge base. That feed is the top production request.

Side finding for regulated processes: on an independent public benchmark of procedure-critical service conversations (SOPBench, 830 tasks), the same platform's strict mode improved task success by 13–32 points and reduced skipped mandatory verification steps by 57–93%. Where a process is compliance-critical (payments, security, escalations), that strict mode — not the conversational mode measured above — is the right configuration, and both run on the same platform per procedure.

Our recommendation

Recommendations

  1. Go to a production pilot with the SOPilot configuration — mined procedures + Context Graph knowledge + automatic routing, in the fast conversational mode measured above. It is the only configuration that combines near-best answer quality, the best response time, and operational visibility (who asked what, what was answered from which source, every routing decision — all auditable per conversation).
  2. Request the live data feed from AENA now. Counter assignments and flight status are the measured #1 remaining quality lever; the integration slot is built and waiting. This single feed converts the most common class of partial answers into complete ones.
  3. Pilot the voice channel on the lost-luggage procedure. The platform's live voice channel is built and tested; lost luggage is the most emotional call type and the procedure where a calm, step-following agent most visibly outperforms.
  4. Keep strict procedure mode for compliance-critical flows. The same platform runs each procedure in the mode that fits it: conversational mode for information delivery (measured here), strict step-enforced mode where regulation demands it (proven on the public SOPBench benchmark).
  5. Let the learning loop run — and keep tests out of it. The agent improves automatically on real traffic (the loop above); run future tests and experiments in a separate workspace so simulated conversations never train the production agent. The platform isolates workspaces natively.
  6. Scale by recipe, not by rewrite. New topics follow the same validated pipeline: recordings → mined procedure → facts into Context Graph → measured against real conversations before going live.
The path to production

Next steps

  1. Live data feeds from AENA. The single biggest quality lever left: today's counter assignments and flight status live on the airport's screens, not in any static knowledge base. The agent has a ready connection slot for this feed; with it, answers like "counter 343 for your flight, today" become possible.
  2. Canonical location data. Desk and office locations change over months (our recordings show it); AENA's floor data would replace recollection with ground truth.
  3. Voice pilot. The platform's live voice channel is built and tested; the natural first showcase is the lost-luggage procedure — the most emotional call type, where a calm, procedure-following agent shines.
  4. Expand beyond the top four topics using the same recipe: mine procedures from recordings, load facts into Context Graph, measure against real conversations before going live.
