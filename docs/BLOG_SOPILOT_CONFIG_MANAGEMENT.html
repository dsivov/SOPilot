Feature discussion · configuration management · SOPilot fit

A Configurator in Disguise

PolarTie wants to kill manual setup: an admin defines templates, a dependency graph, and validation rules; an end-user is then walked — with LLM help — through a guided, validated configuration that can never leave the valid region; all of it visualised and versioned. Read that description again and squint: it is SOPilot's own engine, pointed at a new domain. This doc maps the feature onto what SOPilot already is, names the honest gaps, and proposes how we help.

The ask

What PolarTie asked for

Today a PolarTie instance is configured by hand — prompts, permissions, and (the sharp edge) the tools and MCP servers an agent can use, which customers wire up manually and often get wrong. The proposal is a configuration-management feature that removes the manual toil and the foot-guns, in two LLM-assisted stages plus a visualisation layer.

stage 1 · admin

A PolarTie engineer defines, with LLM help, the templates, the possible configurations, the dependency graph, the validation rules, and the roles/permissions. Constraint-based: the admin sets the rules, everything downstream must obey them.

stage 2 · user

The end-user is guided through setup within those rules — pick a template or customise, with real-time feedback on conflicts, and a view of how one choice affects others. No deep knowledge required; the system keeps them valid.

stage 3 · see

Visualisation & reporting: config status, change history over time, dependency/relationship views, reports. Transparency into what is set and why.

Two cross-cutting constraints: every stage is chat-mode LLM-assisted (ask questions, get guidance, move back and forth iteratively), and it must serve two deployment shapes — a dedicated customer (their own instance, heavy customisation) and a SaaS customer (shared service, limited customisation). The open question they put to us: is SOPilot the right tool, do we extend it, or do we build something new that reuses its parts?

The realization

They described SOPilot

Strip the domain words and the feature is a precise description of what SOPilot already does for conversation procedures. SOPilot is, underneath, a constraint-graph engine with LLM-assisted authoring, validation, versioning, multi-tenancy, and a studio UI. Swap "conversation procedure" for "instance configuration" and the machine is the same one.

An SOP is a graph of steps with dependencies and constraints, authored by an expert with LLM help and then run by a non-expert while the engine keeps them inside the valid region. That is also the exact definition of guided, constraint-based configuration.

This isn't a loose analogy — the correspondence is structural, primitive by primitive (§5). And there's a recursive neatness to it: SOPilot already configures PolarTie agents. The AENA voice agent's procedures, prompt blocks, and — via the connector model and the MCP work we just shipped — its tools and MCP servers are SOPilot artefacts today. This feature is that capability, generalised into a product: SOPilot as the configuration plane for PolarTie itself.

Ground truth — please confirm

Our reading of PolarTie's system

Before designing, we read the pt-multirobot repo so this targets your real architecture, not a guess. Here's what we understand — flagged for your confirmation, so we don't miss anything.

How a robot is configured today

◆ Where the prompts live — and the real pain
The prompts a PolarTie user configures all sit in the same config.json: the freeform prompt (the main system prompt), plus every tool's description, the events messages, and each transfer topic's prompt. That freeform prompt is PolarTie's original, default prompt-management system — SOPilot's structured SOPs are an optional feature on top, so most robots keep the freeform prompt and our product must manage and validate that first, with SOPs opt-in. And the sharpest pain PolarTie engineers named is logical prompt validation: does a freeform prompt make sense, and is it consistent with the rest of the config? — the prompt promises to email the caller but send_email is off/broken; it says "answer in Spanish" but the language/voice disagree; it references a knowledge base or transfer that isn't configured. That's an LLM-checked, semantic validation (the soft-constraint side of §7A) — and it may be the single most valuable thing we deliver, independent of whether the customer uses SOPs at all.

The reframe

Configuration as a guided procedure

The unlock is to treat the configuration process itself as a procedure. The admin authors a config model (nodes = configuration decisions; edges = dependencies; constraints = validation rules; the connector catalog = the available tools). The user then runs that model in a guided wizard, and the output isn't a resolved dialogue — it's a validated configuration artefact that gets applied to their instance.

Read the diagram as effort: the green band and most of the purple and amber already exist. The genuinely new work is a stateful guided-wizard runtime (blue), plus four extensions (templates, admin-authored rules, a typed tool catalog, and an apply step) — the config-domain layer of §6.

The delta

The config-domain layer

Collected, the gaps form one coherent layer on top of the SOPilot engine. Six pieces:

1 · typed config artefact

SOPilot assembles conversation state; here we assemble a typed configuration object with fields, values, and referential integrity. It's the same "typed object → validate → immutable version" loop the SOP definition already uses (TaskDefinition), just a config schema instead of a dialogue schema.

2 · admin-authored constraint rules

Lift validation from hard-coded checks to a data-driven rule set the admin writes — enum / requires / conflicts (§7A). Grounded in real PolarTie foot-guns: email tools require notification_service_url; a LightRAG knowledge base requires its Postgres config; a tool requires the backend it depends on. The collect-problems→publishable→gate pattern stays; the rules become content, not code — and catch the silent missing-tool failures today's config can't.

3 · tool / MCP catalog

Two sources, one picker: PolarTie's built-in catalog from defaults.json (39 tools with their tunable descriptions), plus external tools derived live — point our MCP client at a server and list_tools yields its tools + input schemas. The wizard validates a tool's config against the real schema at author time, instead of the silent missing-tool failure PolarTie has today.

4 · guided-wizard runtime

The one substantially new service: a stateful, resumable guided session that walks the user through the config model, uses allowed-next + the rules to gate each step, gives live feedback, and lets them move back and forth — all LLM-chat-driven.

5 · RBAC & policy envelope

Enforce the dormant ApiKey.role, add a user identity for attribution, and model the policy envelope — the admin-set bounds the user's choices must stay within. This is what makes "admin stage constrains user stage" real.

6 · config bundle + change audit

Near-term output is a validated config.json in exactly PolarTie's format — the same file their Portal writes to robots/<id>/ S3 today. We hand it off; every mutation emits an audit row with attribution (copying the SopVersion pattern). later a first-party apply = an atomic, validated write to that S3 prefix (effect picked up on the next call), with dry-run/rollback — a planned phase after PolarTie's feedback.

◆ The shape of the estimate
Of the whole feature, the engine (graph, constraints, validation-gate, versioning, chat-authoring, connectors, studio, tenancy) is reuse. The config-domain layer above is the real build — and it is well-bounded: a typed artefact, a rules format, a tool catalog, a wizard runtime, RBAC, and an apply path. None of it fights the existing architecture; all of it extends patterns already in the codebase.
Forks in the road

Two design decisions

A. Formal constraints vs LLM-checked constraints

The feature says "LLM assistance" everywhere, but constraint checking has two flavours and we should be deliberate. Hard, structural constraints (dependency ordering, reachability, required-tool-present, type/enum validity) want a formal check — fast, deterministic, and exactly what SOPilot's graph + lint already do. Soft, semantic constraints ("this prompt contradicts that policy", "these two tools overlap confusingly") want the LLM, reading admin rules written in plain language. SOPilot already blends both — a formal graph plus LLM classification — so the recommendation is a hybrid: formal engine for what must be provably correct, LLM for what's judgemental, and the admin authors rules in whichever register fits. Don't try to make the LLM a solver, or the solver a critic.

◆ Decided: feature-model rules
We're holding the admin rules to enum / requires / conflicts — not value arithmetic. That is squarely the formal side and mostly already built: SOPilot's graph is exactly a requires engine (prerequisites); we add conflicts (mutual-exclusion edges) and enum option sets, and reuse the lint-gate to report violations. The LLM layer stays for the soft, semantic checks on top. Bounding expressiveness here is what keeps this a tractable extension rather than a constraint-solver project.

B. Reuse the SOP model literally, or abstract the engine

Tempting shortcut: model a config as an "SOP" with agent_actions and user_states. Resist it — configuration isn't a dialogue, and the vocabulary would leak awkwardly. Better to abstract the engine (graph, constraint evaluation, lint-gate, versioning, chat-to-build) away from the conversation-specific node types, then instantiate it with a config node kind. It's more work than a literal reuse, but it's the difference between a clean second product on a shared core and a conversation tool wearing a config costume.

Two shapes

Dedicated vs SaaS

The two deployment types map cleanly onto SOPilot's tenancy — with one layer to add.

dedicated

A single-tenant instance (SOPilot already runs this way via env — SOPILOT_API_KEY + one project, the same shape as the mounted MCP surface). Templates and rules are tenant-owned and can be deeply customised; the admin stage is that customer's own engineer.

saas · later

Multi-tenant on the Tenant → Project spine (per-tenant Redis quotas already enforced). Templates and rules are platform-level, owned by a PolarTie expert per use-case group; a tenant customises only within that envelope — "admin bounds the user" one level up.

◆ Decided: dedicated first
We start with the dedicated shape — single-tenant, the customer's own engineer as admin, which SOPilot already runs (single-tenant via env). That defers the harder SaaS questions (template scope, the policy envelope owned by a PolarTie expert per use-case group, multi-user RBAC granularity) until dedicated is proven and we've folded in PolarTie's feedback. The data spine already supports both shapes, so this is a sequencing choice, not a fork.
The call

Verdict & recommendation

Yes — SOPilot is the right foundation, and the answer to "extend or build new?" is both, precisely split: reuse SOPilot's engine as a shared core, and build a new configuration product surface (a config-domain layer) on top of it. Not a from-scratch build (it would re-implement a graph, a constraint checker, versioning, an LLM authoring loop, a connector model, and a studio that already exist and are proven on a real customer). Not a literal cram into the SOP model (configuration isn't a conversation).

◆ Why SOPilot specifically, over a generic configurator
Off-the-shelf product configurators and feature-model tools solve the constraint half — but none of them are LLM-native at both ends (chat-to-build for the admin, chat-guided for the user), and none already sit inside the platform that manages the very things being configured: PolarTie's prompts, connectors, and MCP tools. SOPilot's differentiator is exactly the two things this feature is made of — guided-by-LLM and integrated with the agent config. We'd be extending the tool that already configures the agent, not bolting a configurator beside it.
◆ The honest risks — and how round-1 already shrank them
Scope creep into a second product stays the headline risk: this is a product, not a feature — but the round-1 calls cut it hard. Constraint expressiveness → contained: bounded to enum/requires/conflicts (§7A), no half-solver. Apply → parked: deferred to a later phase (P4) after feedback, so we don't build into a live instance blind. SaaS/RBAC → sequenced out: dedicated-first means a single admin, not a multi-user policy engine, for now. What remains genuinely new and worth watching: the stateful guided-wizard runtime (the one net-new service) and, when apply lands, its dry-run/rollback safety.
Sequencing

Build sketch & open questions

Phased so each step is demoable and the riskiest new piece (the wizard runtime) lands on hardened plumbing. Crucially, the visualisation comes first, not last.

◆ Visualisation is the proof — and we've already built it, on your real config
The fastest way to prove this direction is to show it, so we did. The screenshot below is a real, working Config viewer tab in SOPilot's existing Studio, rendering PolarTie's actual AENA robot config.json: its 26 enabled tools grouped by category, both MCP servers with the tool counts we got by introspecting them live (list_tools), and validation panels below. Built almost entirely from parts we already had (net new: a config→graph transform, a validator, one SVG component) — it typechecks, bundles, and renders today, before a line of the wizard. Two load-bearing moments visible in it: the viewer flags two knowledge tools that are switched on with no backend (they'd silently never load), and — the exact thing your team said you "often get wrong" — it checks the prompt's mcp_* references against the tools the MCP servers actually provide.

What this means

Three things, for the PolarTie team reading this:

1. The thesis is proven, not hypothetical. This is a live tab in the SOPilot Studio, built in days, mostly from parts we already had. It reads a real PolarTie config.json and answers the question an engineer can't answer today at a glance — what is this robot actually configured to do, and does it hang together? Every tool, MCP server, knowledge base and transfer topic, plus the backend each one depends on, as a single graph.

2. It catches the failures you ship silently — before a call runs. On your real AENA config it already found two: the built-in knowledge_base_query and knowledge_base_query_lightrag tools are switched on with no backend configured, so they'd never resolve (knowledge here actually comes from MCP). And — the check your team said you "often get wrong" — the viewer runs list_tools against the customer's MCP servers and compares the prompt to what they actually provide: AENA's mcp_search_flights, mcp_ask and mcp_render_page all map to real tools (✓), while 6 more tools the servers expose aren't mentioned in the prompt, so the agent never uses them. If a prompt referenced a tool no server provided, it would flag red. This works with or without SOPs, and it's the same mechanism that would keep the prompt and the tools in sync as either changes.

3. It's the foundation the rest sits on. This viewer is read-only — but the admin templates, the guided user wizard, the constraint rules and RBAC all build on exactly this graph + validation core. Even on its own, as a read-only "X-ray" of a robot's config, it's immediately useful: it turns "why is this tool missing on the call?" from a debugging session into a red edge you see before publishing.
